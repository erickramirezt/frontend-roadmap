accesibility role, name and value

why

user interface components need a role, a name and sometimes a value, to ensure that people using assistive technologies are able to use them. examples of assistive technologies are screen readers, switch controls and speech recognition software

what

there are two cases where we can't use a good html element with built-in accesibility features. event though we want to

  • there is no native html element for what we are trying to achieve
  • there are technical limitations that prevents us using the semantically correct element

how

how do we make sure that custom components have a role, a name and a value?

role

in our last section, buttons a links, we learned that dropdown menu button should be coded as a button. what if our framework does not allow us to do that? if it forces us to use an a instead? if the navegation component in the library we are using, is built with as? then we need to add a role

this is done with the role="button" attribute. now users of assistive technologies can understand what the custom control is. a button has the role="button" built in, so to write button role="button" is redundant

name

the custom control needs a name. in out example, the name is the content of the element, company. as long as we have written our elements like div role="button" Company div, we have a good name. this is also known as the accesible name. the accesible name for our div is company. good

in the following login form, we have several components - a logo, a heading, a label, a dropdown, an input and a button

we are taking a closer look at the label, dropdown and the input. visually there is no clear distinction between the dropdown and the input. the dropdown is coded with a select, which is a correct element for this case. however, it has no name

it has a name attribute. this is not the same as an accesible anme. the name attribute is for computers. in a form, it is used as a reference when the data is submitted. this name countryCode will not help any users. it will not be picked up by assistive technologies

to give this select an accesible name, we must use the attribute aria-label. normally, we would have connected a visual label to the select component. in this case, there is only one visual label fot both the components

value

some component have a value or a state. an accordion is open or closed. this informatino has to be accesible

an accordion is considered a custom component. there is no standard html element to use here. each accordion header should be a button or role="button"
